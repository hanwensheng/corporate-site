## High-Value Paths
- `pages/`: file-based routes such as `index.vue`, `about.vue`, `products.vue`
- `components/home/`: homepage sections like hero, stats, feature grid, CTA
- `components/layout/`: shared shell pieces like `AppHeader.vue`, `AppFooter.vue`
- `components/ui/`: small reusable presentational building blocks
- `composables/`: reusable Nuxt/Vue logic like `useSeo.ts`
- `assets/css/tokens.css`: design tokens via CSS variables
- `assets/css/global.css`: reset, utilities, common layout helpers
- `nuxt.config.ts`: app-level Nuxt, SEO, module, and TypeScript settings
- `uno.config.ts`: UnoCSS presets, shortcuts, breakpoints, and theme extensions

## Commands
### Install
- `npm install`: install dependencies
- `npm run postinstall`: run Nuxt prepare explicitly if `.nuxt/` needs regeneration

### Local Development
- `npm run dev`: start local dev server
- `npm run preview`: preview the built app

### Build
- `npm run build`: production build
- `npm run generate`: generate static output for deployment/prerender

### Lint And Format
- `npm run lint`: run ESLint (flat config via `eslint.config.mjs`, auto-generated by `@nuxt/eslint`)
- `npm run lint:fix`: run ESLint with `--fix` to auto-correct fixable issues
- There is currently no `npm run format` script
- There is currently no Prettier config; rely on ESLint for code consistency
- Run `nuxt prepare` after changing `nuxt.config.ts` modules to regenerate `eslint.config.mjs`

### Tests
- `npm test`: run all Vitest tests once
- `npm run test:watch`: run Vitest in watch mode for development
- `npm run test:coverage`: run tests with V8 coverage
- Tests live under `tests/`, mirroring source structure, e.g. `tests/composables/useSeo.test.ts`
- Test files use `*.test.ts` naming convention
- Vitest uses `happy-dom` as the environment; import `{ mount }` from `@vue/test-utils` for component tests

### Single Test Execution
- Run a single test file: `npx vitest run tests/path/to/file.test.ts`
- Run tests matching a name pattern: `npx vitest run -t "test name pattern"`
- Run a single test in watch mode: `npx vitest tests/path/to/file.test.ts`

## Verification Expectations
- For code or content changes, run `npm run lint` and `npm run build` before claiming completion
- For changes to `composables/` or `utils/`, also run `npm test`
- If a command cannot be run, state that clearly and explain why
- Treat `.nuxt/`, `.output/`, `eslint.config.mjs` (auto-generated), and `node_modules/` as generated artifacts; do not edit them manually

## Architecture Notes
- Pages compose section components rather than embedding large blocks of markup inline
- Shared layout pieces live under `components/layout/`
- Shared presentational primitives live under `components/ui/`
- Reusable cross-page logic belongs in `composables/`
- Styling is split between UnoCSS utility classes and global CSS variables/tokens
- SEO is handled with Nuxt head APIs like `useSeoMeta()`

## Vue And Nuxt Conventions
- Use `<script setup lang="ts">` in Vue SFCs
- Prefer Nuxt auto-imported composables and Vue APIs instead of redundant manual imports
- Use `withDefaults(defineProps<...>(), { ... })` for typed props with defaults
- Keep page components focused on composition and page-level SEO
- Move repeated sections into components instead of duplicating markup across pages
- Prefer `NuxtLink` for internal navigation

## Imports
- Rely on Nuxt auto-imports for `ref`, `computed`, `onMounted`, `onUnmounted`, `useSeoMeta`, `useRuntimeConfig`, etc.
- Do not add manual imports for APIs that Nuxt already auto-imports unless a special case requires it
- Use project-root aliases like `~/` for explicit imports when needed
- Keep imports grouped by source and avoid duplicate import paths for the same module
- Since many components are auto-registered, prefer template usage without local registration boilerplate

## Formatting
- Match the existing style: two-space indentation in Vue, TS, and config files
- Keep trailing commas where the existing file style uses them
- Prefer single quotes in TS/config files to match current repo conventions
- Keep template attributes readable; split across lines when a tag becomes crowded
- Keep objects and arrays vertically formatted once they span multiple properties/items
- Preserve concise Chinese comments already used across the codebase

## TypeScript Guidelines
- TypeScript strict mode is enabled in `nuxt.config.ts`; write code that satisfies strict typing
- Avoid `any`; prefer exact interfaces, unions, or `unknown` with narrowing
- Type props explicitly and keep prop types near the component when only locally used
- Use `interface` for structured object shapes already shared conceptually in a file
- Keep composable function arguments and return values typed
- Watch for Nuxt head/meta typings; use allowed literal unions instead of broad `string` when required

## Naming Conventions
- Vue components: PascalCase filenames, e.g. `HeroBanner.vue`, `AppSectionTitle.vue`
- Component usage follows Nuxt auto-registration prefixes, e.g. `<HomeHeroBanner />`
- Composables: camelCase with `use` prefix, e.g. `usePageSeo`
- Stores: `useXxxStore` naming if/when Pinia stores are added
- Utility functions: descriptive camelCase verbs
- CSS custom properties: kebab-case with `--` prefix
- UnoCSS shortcuts: kebab-case names like `btn-primary`, `section-title`

## Styling Guidelines
- Prefer UnoCSS utility classes for component-local styling
- Reuse shortcuts from `uno.config.ts` before introducing long repeated class lists
- Use CSS variables from `assets/css/tokens.css` for color, spacing, typography, radius, and shadows
- Avoid hardcoding brand colors when an existing token exists
- Keep layout widths aligned with the shared `.container` pattern
- Preserve the current visual language unless the user explicitly asks for a redesign

## SEO And Content
- Every page should define page-specific SEO metadata with `useSeoMeta()` or the shared SEO composable
- Keep metadata and UI copy in Chinese unless the user requests another language
- Use realistic placeholder copy sparingly and mark business placeholders clearly
- Preserve `lang: 'zh-CN'` and other global head defaults unless requirements change

## Error Handling
- This repo currently has little runtime error infrastructure; keep error handling simple and explicit
- For async data later, prefer Nuxt-native patterns such as `useFetch`/`$fetch` with clear fallback UI
- Fail fast on invalid local assumptions instead of silently swallowing errors
- Do not add broad `try/catch` blocks unless they provide user-facing recovery or logging value
- When introducing API layers, centralize repeated error mapping instead of duplicating ad hoc handling

## File Placement Rules
- New route = add a new file under `pages/`
- New reusable section for homepage = add under `components/home/`
- New global shell/shared frame element = add under `components/layout/`
- New small cross-page presentational piece = add under `components/ui/`
- New reusable business logic = add under `composables/`
- New static asset = add under `public/` unless it must be processed by the build pipeline
